========================================================================
             영화 시맨틱 검색 엔진 프로젝트 기반 기술 명세서 (Tech.txt)
========================================================================

이 프로젝트는 대규모 언어 모델(LLM)의 임베딩 모델(Embedding Model)과 로컬에 구축된 관계형 
데이터베이스(Supabase/PostgreSQL) 및 벡터 연산 확장팩(pgvector)을 결합하여, 서버 비용을 
최소화하면서 엔터프라이즈급의 데이터 확장성과 고속 검색 성능을 구현한 '하이브리드 시맨틱 검색 시스템'입니다.

1. 데이터 수집 및 구조화 기술 (Data Collection & Structuring)
------------------------------------------------------------------------
* 사용 모델: Gemini 3.1 Flash Lite (gemini-3.1-flash-lite)
* 핵심 기술: Structured Outputs (구조화된 출력)
  - Python Pydantic 라이브러리를 활용하여 출력 스키마(Schema)를 강제 지정.
  - API 반환값의 JSON 포맷 유효성 및 데이터 정합성(제목, 연도, 배우, 줄거리)을 100% 보장.
* 수집 알고리즘: 중복 검사 루프 (Incremental Loop)
  - 기존 수집 데이터를 제외 대상으로 지정하여 API를 중복 호출하지 않도록 자동 요청.
  - 카테고리당 정확히 고유 데이터 100개씩(총 300개) 수집될 때까지 반복 확보.

2. 의미론적 벡터 임베딩 기술 (Semantic Embedding)
------------------------------------------------------------------------
* 사용 모델: Gemini Embedding v2 (gemini-embedding-2)
* 출력 차원: 3,072차원 고정밀 밀집 벡터 (Dense Vector)
* 문서 가공: 영화 메타데이터(제목, 개봉연도, 배우)와 서사(줄거리)를 한 문단으로 
  결합하여 문맥적 의미가 보존된 문서(Document) 형태를 제작해 임베딩 진행.
* 최적화: 증분 임베딩 (Incremental Caching)
  - 신규 데이터 추가 시, 이미 생성된 벡터 연산은 재사용하고 새 영화만 선별하여 
    임베딩 연산을 수행하므로 API 할당량(Quota) 및 대기 시간 극단적 절약.

3. 데이터베이스 및 벡터 검색 아키텍처 (Database & pgvector Search Engine)
------------------------------------------------------------------------
* DB 엔진: 로컬 Docker 기반 Supabase (PostgreSQL 15+)
* 벡터 확장팩: pgvector
  - 3,072차원의 고밀도 벡터 데이터를 관계형 테이블(`movies` 테이블) 내에 안전하게 저장.
* 검색 알고리즘: DB 레벨 코사인 유사도 연산 (match_movies RPC 함수)
  - 사용자가 입력한 자연어 쿼리를 임베딩하여 DB로 전송하면, 데이터베이스 엔진 내부에서 
    직접 코사인 유사도 공식(1 - (embedding <=> query_embedding))을 계산.
  - 쿼리와 데이터베이스 간의 연산을 C언어 네이티브 수준으로 실행하여 극도로 빠른 
    응답속도(수 ms 내외)를 보장.

4. 2D 임베딩 공간 시각화 기술 (2D Embedding Visualization)
------------------------------------------------------------------------
* 분석 알고리즘: 주성분 분석 (PCA - Principal Component Analysis)
* 차원 축소: 300개 영화의 3,072차원 고차원 벡터를 2차원 평면 좌표(X, Y)로 압축 투영.
* 코사인 거리 보존 (L2 Normalization):
  - PCA 투영 전, 각 임베딩 벡터를 L2 정규화하여 벡터 길이를 1로 통일.
  - 이를 통해 2D 평면상에서의 유클리드 거리가 실제 코사인 유사도와 수학적으로 정비례하도록 보정하여 시각화 왜곡을 완벽히 방지.
* 렌더링: Chart.js (2D Scatter Plot)
  - 카테고리별 독립적인 PCA 연산 및 개별 차트 제공을 통해 3개 장르의 군집 분포도를 개별 렌더링.

5. 이전 방식(로컬 파일 시스템) vs 현재 방식(Supabase DB) 성능 및 아키텍처 비교
------------------------------------------------------------------------

| 비교 항목 | 이전 방식 (NumPy 파일 시스템) | 현재 방식 (Supabase + pgvector DB) |
| :--- | :--- | :--- |
| **저장 스택** | JSON + NumPy `.npy` 로컬 이진 파일 | **Docker 상의 PostgreSQL 관계형 DB** |
| **검색 연산 주체** | 웹 서버 (Python NumPy 모듈 호출) | **데이터베이스 엔진 (PostgreSQL pgvector)** |
| **서버 메모리(RAM)**| **낮은 효율**: 모든 벡터 데이터를 상시 RAM에 상주 | **높은 효율**: 메모리 상주 없이 필요시 인출 및 인덱싱 |
| **데이터 확장성** | **한계 존재**: 데이터 수만 건 증가 시 서버 폭사 | **무제한**: 수백만 건 데이터도 디스크와 인덱스로 거뜬히 감당 |
| **동시성 & 트랜잭션**| 파일 락 관리 불가, 다중 쓰기 시 유실 위험 | **안전**: RDBMS의 ACID 트랜잭션 및 안전한 쓰기 보장 |
| **비즈니스 로직 분리**| 파이썬 서버 코드 내에 유사도/필터링 로직 혼재 | **DB RPC 함수(`match_movies`)**에 검색 로직 캡슐화 |
| **추가 인프라 비용** | 없음 (파이썬 구동만으로 해결) | Docker Desktop 구동 필요 및 로컬 자원 사용량 증가 |

* **종합 평가**:
  - **이전 파일 방식**은 로컬 자원의 메모리를 모두 사용하여 300개 수준의 아주 작은 데이터셋에서만 초고속으로 작동하는 단발성 장난감 아키텍처였습니다.
  - **현재 Supabase DB 방식**은 엔터프라이즈 레벨의 안정성, 무제한적인 데이터 확장성, 트랜잭션 안전성 및 다중 사용자 동시 접속을 보장하는 상용 서비스 규격의 구조입니다.

6. 제공 API 명세 요약 (REST API via Python http.server)
------------------------------------------------------------------------
* GET /api/movies?category={category}: DB 내 해당 카테고리 영화의 가벼운 메타데이터 일괄 조회.
* GET /api/search?q={query}&category={category}: Supabase RPC(match_movies)를 호출하여 시맨틱 검색 결과 Top 3 고속 반환.
* GET /api/visualization: DB 내 300개 영화 임베딩의 수치 리스트를 받아와 PCA 좌표 계산 후 2D 데이터 일괄 반환.

7. 프리미엄 웹 인터페이스 (Frontend UI/UX)
------------------------------------------------------------------------
* 기술 스택: Vanilla HTML5, CSS3, Javascript (ES6+)
* 디자인 테마: 다크 모드 기반의 글래스모피즘 (Glassmorphism)
* 인터랙션:
  - 카테고리 전환 시 즉각적인 탭 토글 및 카드 페이드인(Fade-in) 애니메이션.
  - Chart.js 마우스 호버(Hover) 시, 해당 좌표 영화의 제목, 개봉 연도, 출연자, 상세 줄거리 요약이 미려한 반투명 다크 툴팁으로 실시간 시각화.
